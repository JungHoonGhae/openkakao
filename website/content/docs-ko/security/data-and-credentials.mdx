# 데이터와 자격 증명

이 페이지는 실무적인 질문 하나에 답합니다. OpenKakao는 실제로 어떤 데이터를 만지는가?

## 로컬 읽기

명령에 따라 OpenKakao는 다음을 읽을 수 있습니다.

- 로그인 추출에 필요한 KakaoTalk 앱 캐시 데이터
- 인증 요청을 재구성하는 데 필요한 로컬 식별자와 요청 메타데이터
- `~/.config/openkakao/credentials.json`에 저장된 기존 자격 증명

## 로컬 저장

CLI는 다음 위치에 자격 증명을 저장할 수 있습니다.

```text
~/.config/openkakao/credentials.json
```

이 파일은 현재 사용자만 읽고 쓸 수 있도록 `0600` 권한으로 저장합니다.

## 네트워크 대상

CLI는 데스크톱 앱이 사용하는 Kakao 인프라와 통신하도록 설계되어 있습니다.

- `katalk.kakao.com`
- `talk-pilsner.kakao.com`
- `booking-loco.kakao.com`
- `ticket-loco.kakao.com`
- 동적으로 할당되는 LOCO 서버
- 미디어 흐름에 사용되는 Kakao CDN 엔드포인트

## 기본 모델에 없는 것

기본적으로 OpenKakao는 다음을 도입하지 않습니다.

- OpenKakao cloud sync 서비스
- 메시지에 대한 원격 프로젝트 telemetry
- 계정 트래픽을 프록시하는 프로젝트 호스팅 API
- 채팅 히스토리를 제3자 서비스로 자동 export 하는 기능

## 사용자의 책임

JSON을 다른 도구로 넘기는 순간 프라이버시 모델은 바뀝니다.

예시:

- 채팅 데이터를 외부 LLM API로 보내는 것은 사용자의 결정
- export를 공유 폴더나 데이터베이스에 저장하는 것도 사용자의 결정
- 메시지를 webhook이나 bot으로 전달하면 즉시 리스크 표면이 바뀜

OpenKakao는 기본 모델을 local-first로 유지합니다. 사용자의 자동화 스택은 그렇지 않을 수 있습니다.

## Hook와 Webhook 경계

`watch --hook-cmd`는 같은 머신의 셸 명령에 정규화된 JSON을 넘기므로 이벤트 처리를 로컬에 유지합니다.

`watch --webhook-url`은 경계를 바꿉니다.

- 매칭된 이벤트 payload가 HTTP로 외부 전송됨
- 목적지의 보관 정책이 중요해짐
- receiver가 신뢰 모델의 일부가 됨

`--webhook-signing-secret`를 켜면 요청에 다음 헤더가 추가됩니다.

- `X-OpenKakao-Timestamp`
- `X-OpenKakao-Signature`

이것은 receiver가 무결성을 검증하는 데 도움을 주지만, 전송 보안, timestamp 검증, receiver 측 접근 제어를 대신하지는 않습니다.
