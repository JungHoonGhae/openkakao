# 인증

OpenKakao는 실행 중인 KakaoTalk macOS 앱에서 인증 재료를 가져오고, 사용자가 요청했을 때만 재사용 가능한 자격 증명을 로컬에 저장합니다.

## 토큰 종류

| 자격 증명 경로 | 출처 | 적합한 용도 |
|---|---|---|
| saved credentials | `~/.config/openkakao/credentials.json` | 일반적인 CLI 시작 |
| Cache.db extraction | 최근 KakaoTalk macOS 요청 상태 | 저장된 상태가 낡았을 때의 fallback |
| `login.json` relogin | 데스크톱 로그인 흐름 재구성 | LOCO 계열 명령 복구 |
| `refresh_token` renewal | 갱신 엔드포인트 + 캐시된 refresh token | 감독된 토큰 갱신 시도 |

## 전송 계층 경계

- REST: 프로필, 친구, 설정, 최근 연 채팅, 캐시된 메시지
- LOCO: 전체 채팅 목록, 전체 메시지 히스토리, 전송, watch, 미디어 흐름

안정적인 자동화가 중요하다면 REST는 싼 health check에 쓰고, 실제 채팅 작업은 LOCO로 처리하세요. 자세한 기준은 [REST vs LOCO](/ko/docs/getting-started/transport-boundary)에 정리했습니다.

## 초기 로그인

```bash
openkakao-rs login --save
```

이 명령은 필요한 로컬 입력을 추출하고 로그인한 뒤 다음 위치에 자격 증명을 기록합니다.

```text
~/.config/openkakao/credentials.json
```

파일 권한은 `0600`으로 생성됩니다.

## 상태 점검

```bash
openkakao-rs auth
openkakao-rs doctor
```

`auth`는 빠른 REST 측 점검입니다. 더 엄격한 LOCO 경로까지 건강한지 확인하려면 `doctor --loco`를 사용하세요.

## 복구 정책

```bash
openkakao-rs relogin --fresh-xvc
openkakao-rs renew
```

복구 사다리는 이제 정책 기반으로 결정됩니다.

1. 저장된 자격 증명 재사용
2. config에 따라 `relogin` 또는 `renew`를 먼저 시도
3. 활성화된 나머지 복구 경로 시도
4. fresh Cache.db extraction으로 fallback

기본 동작은 여전히 보수적입니다.

- `auth.prefer_relogin = true`
- `auth.auto_renew = true`

즉 기본 순서는 다음과 같습니다.

1. saved credentials
2. `login.json` relogin
3. `refresh_token` renewal
4. fresh Cache.db extraction

`auto_renew`를 끄면 renewal 단계는 건너뜁니다. `prefer_relogin = false`로 두면 relogin보다 renewal을 먼저 시도합니다.

upstream 클라이언트나 서버 동작이 바뀐 뒤에도 복구가 항상 성공한다고 가정한 unattended job을 만들면 안 됩니다.

## Unattended 모드

사람 확인 없이 side effect를 만들 수 있는 명령은 전역 opt-in이 필요합니다.

- `send -y`
- `send-file -y`
- `send-photo -y`
- `watch --read-receipt`
- `watch --hook-cmd`
- `watch --webhook-url`

예시:

```bash
openkakao-rs --unattended --allow-watch-side-effects watch --hook-cmd 'jq . > /tmp/event.json'
```

의도는 단순합니다. CLI가 기다리지 않고 행동한다면, 사용자가 그것을 분명하게 선언해야 합니다.

지속적인 운영에서는 권한과 auth 정책을 [설정](/ko/docs/getting-started/configuration)으로 옮기세요.

## 자주 보이는 실패 코드

| 코드 | 의미 | 일반적인 대응 |
|---|---|---|
| `-950` | 토큰 만료 또는 LOCO 로그인 거부 | relogin부터 수행하고, renewal을 계속 켤지 점검 |
| `-300` | device 또는 요청 불일치 | 식별자, 로컬 앱 상태, 로그인 입력 점검 |
| `-400` | 파라미터 누락 | 로그인 다시 수행, 추출값 검증 |
