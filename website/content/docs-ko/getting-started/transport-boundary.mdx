# REST vs LOCO

OpenKakao는 두 가지 전송 계층을 사용합니다. 둘은 서로 대체 관계가 아닙니다.

## 이런 경우에는 REST를 쓰세요

- 토큰이나 계정 상태를 빠르게 확인할 때
- 프로필, 친구, 설정처럼 계정 인접 데이터를 읽을 때
- 데스크톱 앱이 최근 열어 둔 채팅/메시지 캐시만으로 충분할 때
- 짧은 one-shot 명령에서 세션 부담을 줄이고 싶을 때

REST는 가벼운 경로입니다. 진단과 소규모 읽기에는 유용하지만, 메시징 기준의 기준 경로는 아닙니다.

## 이런 경우에는 LOCO를 쓰세요

- 전체 채팅 목록이 필요할 때
- 서버에 남아 있는 메시지 히스토리가 필요할 때
- 메시지나 미디어를 전송할 때
- 실시간 `watch`가 필요할 때
- KakaoTalk 활성 메시징 세션과 같은 동작을 원할 때

LOCO는 더 무겁지만, 실제 채팅 자동화에서 중요한 경로는 이쪽입니다.

## 실무 경계

| 요구사항 | 권장 경로 | 이유 |
|---|---|---|
| `auth`, `me`, `friends`, `settings` | REST | 더 빠르고 단순한 계정 점검 |
| `chats`, `read` | 캐시로 충분할 때만 REST | 데스크톱 앱이 최근 연 항목에 제한됨 |
| `loco-chats`, `loco-read` | LOCO | 채팅/메시지에 대한 기준 접근 |
| `send`, `send-file`, `send-photo`, `watch` | LOCO | 전송과 실시간 메시징은 여기서 동작 |
| `doctor` | REST 우선, 필요시 LOCO | 인증 문제인지 전송 계층 문제인지 분리하기 좋음 |

## 안정성 가이드

unattended 운영에서는 REST와 LOCO가 같은 방식으로 실패한다고 가정하면 안 됩니다.

- REST는 LOCO 재인증이 필요한 상황에서도 가벼운 계정 점검에 응답할 수 있습니다.
- LOCO는 유효한 live session이 필요하므로 서비스 건강 상태를 더 엄격하게 보여줍니다.
- 메시지 전달이나 전체 히스토리에 의존한다면 LOCO를 직접 검증하세요.
- 로컬 앱 상태가 대체로 건강한지만 보고 싶다면 REST가 더 싼 1차 점검입니다.

## 운영 권장 규칙

1. 상태를 빠르게 확인할 때는 REST를 쓴다
2. 실제 채팅 자동화는 LOCO를 쓴다
3. 서비스 건강 판단에서는 LOCO 실패를 더 강한 신호로 본다
