## 전송 계층 경계

인증 명령은 전송 계층 모델을 염두에 두고 사용해야 합니다.

| 필요 | 권장 경로 | 이유 |
|------|-----------|------|
| 빠른 계정/프로필/친구 점검 | REST | 요청 흐름이 더 단순하고 세션 복잡도가 낮음 |
| 전체 채팅 목록, 전체 히스토리, 전송, watch | LOCO | 서버 보관 데이터와 실시간 메시징이 여기 있음 |
| 토큰 상태 점검 | 우선 REST | 저장된 자격 증명이 아직 쓸 만한지 가장 빨리 확인 가능 |
| LOCO 실패 복구 | config 기준 | 기본은 `relogin` 우선, 필요하면 `renew` 우선으로 조정 가능 |

## auth

현재 토큰의 유효성을 확인합니다.

```bash
openkakao-rs auth
```

**출력 예시:**
```
  User ID: 100000001
  Token:   a1b2c3d4...
  Version: 3.7.0
  Token is valid!
```

## login

실행 중인 KakaoTalk 앱에서 자격 증명을 추출합니다.

```bash
openkakao-rs login --save
```

| Flag | 설명 |
|------|------|
| `--save` | 추출한 자격 증명을 `~/.config/openkakao/credentials.json`에 저장 |

## relogin

저장된 로그인 파라미터를 사용해 access token을 다시 발급받습니다.

```bash
openkakao-rs relogin [OPTIONS]
```

| Flag | 설명 |
|------|------|
| `--fresh-xvc` | X-VC 인증 헤더를 재생성 |
| `--password <PASSWORD>` | 캐시 대신 특정 비밀번호 사용 |
| `--email <EMAIL>` | 저장된 이메일 override |

다음 상황에서 우선 고려합니다.

- LOCO 명령이 인증 오류로 실패하기 시작할 때
- KakaoTalk 앱 업데이트 후 saved credentials가 어긋났을 때
- 데스크톱 앱이 쓰는 `login.json` 경로로 다시 기준을 맞추고 싶을 때

## renew

`refresh_token`을 사용해 토큰 갱신을 시도합니다.

```bash
openkakao-rs renew
```

다음 두 엔드포인트를 시도합니다.
1. `oauth2_token.json` (access_token + refresh_token)
2. `renew_token.json` (refresh_token only)

`renew`는 다음 상황에 맞습니다.

- 이미 refresh token 경로가 살아 있을 때
- 전체 relogin 전에 더 가벼운 재시도를 원할 때
- unattended 영속 성공을 가정하지 않고, 감독된 recovery step을 구성할 때

## 복구 순서

서비스 안정성을 위해 CLI는 recovery 순서를 정책으로 다룹니다.

기본값:

1. saved credentials
2. `relogin`
3. `renew`
4. fresh `Cache.db` extraction

설정에서 다음을 바꿀 수 있습니다.

- `auth.prefer_relogin = false` -> `renew`를 먼저 시도
- `auth.auto_renew = false` -> renewal 단계 제거

실무 경계는 단순합니다. REST와 LOCO는 같은 recovery ladder를 재사용하지만, LOCO는 유효한 live session이 필요하므로 더 엄격한 경로입니다.
