---
title: watch
description: 자동 재연결을 포함한 실시간 메시지 모니터링.
---

`watch`는 이제 OpenKakao 자동화의 이벤트 경계입니다.

- 가능하면 `--hook-cmd`로 로컬에 머무르세요.
- 메시지 이벤트를 머신 밖으로 보내려는 의도가 분명할 때만 `--webhook-url`을 쓰세요.
- 이것을 보장된 큐가 아니라 best-effort 전달로 취급하세요.

## 사용법

```bash
openkakao-rs watch [OPTIONS]
```

## 옵션

| Flag | 설명 | 기본값 |
|------|------|--------|
| `--chat-id <ID>` | 특정 채팅만 필터링 | 전체 채팅 |
| `--raw` | raw BSON body 출력 | false |
| `--read-receipt` | 수신 메시지에 대해 NOTIREAD 전송 | false |
| `--max-reconnect <N>` | 최대 재연결 횟수 (0 = 비활성화) | 5 |
| `--download-media` | 미디어 첨부 자동 다운로드 | false |
| `--download-dir <DIR>` | 다운로드 디렉토리 | `downloads` |
| `--hook-cmd <CMD>` | 매칭된 메시지 이벤트에 대해 로컬 셸 명령 실행 | disabled |
| `--webhook-url <URL>` | 매칭된 메시지 이벤트를 webhook으로 POST | disabled |
| `--webhook-header "Name: Value"` | webhook 요청에 HTTP 헤더 추가 | none |
| `--webhook-signing-secret <SECRET>` | HMAC-SHA256으로 webhook payload 서명 | disabled |
| `--hook-chat-id <ID>` | 특정 chat ID에만 hook 실행 제한 | 모든 채팅 |
| `--hook-keyword <TEXT>` | 특정 키워드가 포함된 메시지에만 hook 실행 제한 | 모든 메시지 |
| `--hook-type <TYPE>` | 특정 메시지 타입 코드에만 hook 실행 제한 | 모든 타입 |
| `--hook-fail-fast` | hook 명령 실패 시 watch 종료 | false |

`--read-receipt`, `--hook-cmd`, `--webhook-url` 중 하나라도 켜면 unattended watch side effect 권한도 같이 허용해야 합니다.

- CLI: `--unattended --allow-watch-side-effects`
- 또는 config: `mode.unattended = true` 와 `watch.allow_side_effects = true`

## 예시

```bash
# 모든 채팅 감시
openkakao-rs watch

# 특정 채팅만 감시하고 읽음 처리
openkakao-rs --unattended --allow-watch-side-effects watch --chat-id 382416827148557 --read-receipt

# 미디어 자동 다운로드
openkakao-rs watch --download-media --download-dir ./media

# 최대한 끈질기게 재연결
openkakao-rs --unattended --allow-watch-side-effects watch --max-reconnect 20 --read-receipt --download-media

# 한 채팅에서 "urgent"가 오면 로컬 명령 실행
openkakao-rs watch \
  --unattended \
  --allow-watch-side-effects \
  --chat-id 382416827148557 \
  --hook-cmd 'jq -r .message | osascript -e "display notification (do shell script \"cat\") with title \"OpenKakao\""' \
  --hook-keyword urgent

# 매칭된 이벤트를 webhook으로 전달
openkakao-rs watch \
  --unattended \
  --allow-watch-side-effects \
  --webhook-url https://example.com/openkakao \
  --webhook-header 'Authorization: Bearer token' \
  --webhook-signing-secret 'super-secret' \
  --hook-chat-id 382416827148557 \
  --hook-type 1
```

## 재연결 동작

| 상황 | 동작 |
|------|------|
| 네트워크 오류 | 지수 백오프 (1s, 2s, 4s, ... 최대 32s) |
| 인증 오류 (`-950`) | 토큰 복구 시도 후 재연결 |
| `CHANGESVR` 수신 | 새 서버로 즉시 재연결 |
| `PING` timeout | 연결 끊김으로 처리 |
| 최대 재시도 초과 | 종료 |

## 출력 형식

```
[Chat Title] Author: message text
```

미디어 메시지는 첨부 타입을 표시합니다.

```
[Chat Title] Author: [Photo] (attached)
[Chat Title] Author: [Video] video.mp4
```

## Hook 이벤트 payload

`--hook-cmd`를 설정하면 매칭된 메시지 이벤트가 `stdin` JSON으로 전달됩니다.

`--webhook-url`을 설정하면 같은 이벤트 JSON이 `Content-Type: application/json`으로 HTTP `POST` 전송됩니다.

`--webhook-signing-secret`를 설정하면 요청에 다음 헤더도 포함됩니다.

- `X-OpenKakao-Timestamp`
- `X-OpenKakao-Signature`

서명 형식은 `sha256=<hex>`이고, `timestamp.payload`를 기준으로 서명합니다.

이것은 receiver에 무결성 점검을 제공할 뿐, replay protection까지 자동으로 해결하지는 않습니다. webhook을 공개로 노출한다면 receiver 측에서 timestamp window도 같이 검증해야 합니다.

다음 환경 변수도 함께 제공합니다.

- `OPENKAKAO_EVENT_TYPE`
- `OPENKAKAO_CHAT_ID`
- `OPENKAKAO_CHAT_NAME`
- `OPENKAKAO_LOG_ID`
- `OPENKAKAO_AUTHOR_ID`
- `OPENKAKAO_AUTHOR_NICKNAME`
- `OPENKAKAO_MESSAGE_TYPE`
- `OPENKAKAO_MESSAGE_TYPE_LABEL`

예시:

```bash
openkakao-rs watch \
  --unattended \
  --allow-watch-side-effects \
  --hook-cmd 'jq . > /tmp/openkakao-event.json' \
  --hook-chat-id 382416827148557 \
  --hook-type 1
```

Webhook 예시:

```bash
openkakao-rs watch \
  --unattended \
  --allow-watch-side-effects \
  --webhook-url https://hooks.example.com/openkakao \
  --webhook-header 'X-OpenKakao-Source: laptop' \
  --webhook-signing-secret 'super-secret' \
  --hook-keyword urgent
```

서명 검증 예시:

```bash
BODY_FILE=/tmp/openkakao-body.json
printf '%s.%s' "$HTTP_X_OPENKAKAO_TIMESTAMP" "$(cat "$BODY_FILE")" \
  | openssl dgst -sha256 -hmac 'super-secret'
```

## 운영 경계

`watch` hook은 의도적으로 단순합니다.

- delivery queue 없음
- webhook 전달에 대한 retry/backoff 정책 없음
- 내장 deduplication 없음
- idempotency key 관리 없음
- 여러 hook 정의를 위한 config file 없음

지금은 여기서 멈추는 것이 맞습니다. 더 강한 보장이 필요하다면, CLI를 이벤트 버스로 가정하지 말고 receiver나 wrapper process에서 보강하세요.
